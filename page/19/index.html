Debian系dpkg和redhat系rpm命令总结

Mar 13th, 2012

前言：最近突然用ubuntu，发现好多包管理命令dpkg的用法很生疏了，在这里正好总结一下，以及和rpm作一些相似作用的列举

1 软件包的信息  将会列出这个软件包的详细资料

dongwm@linux-dongwm:~> rpm  -qpi rubygem-capistrano-2.11.2-1.2.i586.rpm dongwm@dongwm-pc:~$ dpkg-deb -I capistrano_2.5.9-1_all.deb

dongwm@dongwm-pc:~$ dpkg -s  capistrano 或者

dongwm@dongwm-pc:~$  dpkg-query -s capistrano 2 软件包内文件内容  列出安装后的所有文件的全路径

dongwm@linux-dongwm:~> rpm  -qpl rubygem-capistrano-2.11.2-1.2.i586.rpm

dongwm@dongwm-pc:~$ dpkg-deb -c capistrano_2.5.9-1_all.deb |more

或者

dongwm@dongwm-pc:~$ dpkg –contents capistrano_2.5.9-1_all.deb

3 软件包的依赖关系

dongwm@linux-dongwm:~> rpm  -qpR rubygem-capistrano-2.11.2-1.2.i586.rpm

dongwm@dongwm-pc:~$ dpkg-deb -f capistrano_2.5.9-1_all.deb depends //depends Depends: ruby1.8, libnet-ssh2-ruby1.8, libnet-ssh-gateway-ruby1.8, libnet-sftp2-ruby1.8, libnet-scp-ruby1.8, libhighline-ruby1.8

4 数据库查询 当每个软件包安装在Linux系统后，安装文件都会列到数据库

dongwm@linux-dongwm:~> rpm -qa capistrano

dongwm@dongwm-pc:~$ dpkg -l capistrano 期望状态=未知(u)/安装(i)/删除(r)/清除(p)/保持(h) | 状态=未安装(n)/已安装(i)/仅存配置(c)/仅解压缩(U)/配置失败(F)/不完全安装(H)/触发器等待(W)/触发器未决(T) |/ 错误?=(无)/须重装(R) (状态，错误：大写=故障) ||/ 名称         版本         描述 +++-==============-==============-============================================ iU  capistrano     2.5.9-1        Executes commands in parallel on multiples s   #这里的 iu表示想安装，但是只是解压缩没有安装成功  debian系的查询显示比较复杂

5 安装软件包

dongwm@linux-dongwm:~> sudo rpm -ivh rubygem-capistrano-2.11.2-1.2.i586.rpm

dongwm@dongwm-pc:~$ sudo dpkg -i capistrano_2.5.9-1_all.deb

6 软件卸载

dongwm@linux-dongwm:~> sudo rpm -e rubygem-capistrano-2.11.2-1.2

dongwm@dongwm-pc:~$ sudo dpkg -r capistrano

或者

dongwm@dongwm-pc:~$ sudo dpkg –purge capistrano

7 查询文件归属

dongwm@linux-dongwm:~> rpm -qf /usr/bin/vim  查询 /usr/bin/vim  属于那个包

dongwm@dongwm-pc:~$ dpkg –search /usr/bin/mawk

7 查询软件包都安装到了那里

dongwm@linux-dongwm:~> rpm -ql gawk

dongwm@dongwm-pc:~$ dpkg -S mawk

gentoo笔记（十一）Overlay和layman

Mar 7th, 2012

前言：Overlay是包含额外的Gentoo的ebuild的Portage包树，类似于“第三方源”，使用它的原因我以为就是让那些不在Portage包树的软件包，当你修改，开发，测试的时候，不会影响主干包树，直到它稳定到加到主干包树里面。当修改主干包树时候，下一次同步你的修改就没有了，但是Overlay能保存你的自定义修改。layman（英语意为‘门外汉’，意思很明显了把）使用它安装和更新这样的Overlay。他会有自己的一套portage包树，和emerge平齐

1 安装layman

dongwm@localhost ~ $sudo emerge layman 2 指定layman的Portage库

dongwm@localhost ~ $sudo echo “source /var/lib/layman/make.conf” >> /etc/make.conf 3 列出可用的Overlay源

dongwm@localhost ~ $sudo layman -L 其中有一行

* gentoo-china              [Subversion] (http://gentoo-china-overlay.google…)

4 安装Overlay源

dongwm@localhost ~ $sudo layman -a gentoo-china 5 从Overlay源安装一个软件包

因为gentoo-china的软件包目录在/var/lib/layman/gentoo-china  可以去找你想要安装的软件包名称 我这里是/var/lib/layman/gentoo-china/app-editors/uex

dongwm@localhost ~ $sudo emerge uex # emerge会自动找到 6 更新Overlay源

dongwm@localhost ~ $sudo layman -S

7 删除Overlay源

dongwm@localhost ~ $sudo layman -d desktop-effects  删除desktop-effects

   

Opensuse安装anjuta

Feb 27th, 2012

问题：以前使用anjuta在gentoo下调试c/c++程序，最近在opensuse11.4想使用anjuta，发现yast/zypper安装的anjuta不能新建项目，直接卡住，然后杀掉进程后再启动anjuta会停止在”加载调试器”这一步。

解决：1 使用源码安装：http://ftp.gnome.org/pub/GNOME/sources/anjuta

发现各个版本（从2.1.0到3.3）都有一些依赖关系，重新编译安装会对现在的系统造成影响，比如我安装了一个新版本的glib，造成xwindows启动不了，最后在修复模式卸载了才恢复

2  使用jhbuild

提示 在autogen的过程中： gtkdocize: command not found  失败

3 根据anjuta官网wiki https://live.gnome.org/DeveloperTools/Installation 文档安装 还是同样的问题，不能新建项目

最终解决办法：

从github上面找了一个anjuta源码版本：

https://github.com/abderrahim/anjuta.git 过程：

dongwm@linux-dongwm:~> git clone https://github.com/abderrahim/anjuta.git cd anjuta/ ./autogen.sh

make

sudo make install

因为默认安装在/use/local下，需要

sudo vi /etc/ld.so.conf 添加一行： /usr/local/lib

sudo /sbin/ldconfig

这个anjuta正常了

gentoo笔记（九）Portage 与软件包

Feb 24th, 2012

前言：Portage 工具中用户用得最多的便是 emerge，它 是一个用以安装, 卸载, 查询以及维护软件的命令. 它是 ebuild 的一个前端，而且它有ebuild缺少的功能（比如解决依赖性、搜索Portage树等。

1 查看 Portage 树 lls /usr/portage –classify 注，也能切换到更深的目录查看类别，软件包等 2 搜寻软件包

emerge –search mozilla 或者emerge -s mozilla # 搜寻和anjuta有关的软件信息

Searching… [ Results for search key : anjuta ] [ Applications found : 1 ]

*  dev-util/anjuta Latest version available: 2.32.1.1 Latest version installed: 2.32.1.1 Size of files: 9,934 kB Homepage:      http://www.anjuta.org Description:   A versatile IDE for GNOME License:       GPL-2 3 搜寻相关描述信息 dongwm@localhost ~ $sudo emerge –searchdesc mozilla 4 更新portage dongwm@localhost ~ $sudo emerge –sync #要以 rsync 更新 Portage,因为不更新portage树的话 gentoo软件会停留在一些版本上，系统会很快变得过时 5 预编译 也就是假装编译，其实并没有编译安装 dongwm@localhost ~ $sudo emerge –pretend anjuta （不编译安装，可以用来查看这个软件包将安装的依赖性软件，简写: -p） 6 只获取软件的源码 dongwm@localhost ~ $sudo emerge –fetchonly anjuta  [不编译安装，下载这个软件的源代码以及他的依赖性软件，简写：-f] 7 安装指定版本的 dongwm@localhost ~ $sudo emerge “<anjuta-2.22” 8 显示安装软件所有的依赖性 dongwm@localhost ~ $sudo emerge –emptytree –pretend anjuta 9 查看软件所需源代码的网址 dongwm@localhost ~ $sudo emerge –fetchonly –pretend anjuta 10 不理会依赖关系而安装 dongwm@localhost ~ $sudo emerge –nodeps anjuta 11 安装软件的依赖关系 dongwm@localhost ~ $sudo emerge –onlydeps anjuta 12 列出与删除已不需要的 (orphaned) 依赖关系

dongwm@localhost ~ $sudo emerge -p depclean (在认真检查过输出的列表之后, 开始卸载不用的依赖关系包) dongwm@localhost ~ $sudo emerge depclean

注：使用 depclean 很有可能严重地损坏gentoo系统 ，需要小心谨慎, 并在实际删除前反复检查依赖关系包列表. 13 更新系统软件包 dongwm@localhost ~ $sudo emerge –update system 14 更新整个系统 dongwm@localhost ~ $sudo emerge –update world 15 预更新整个系统

dongwm@localhost ~ $sudo emerge –pretend –update world

在单词 “ebuild” 旁边您可以看到一个字母 (或字母的组合)：

B (blocks) 左边列出来的软件因为冲突原因将阻碍右边列出来的软件的安装

N (new) 对于您的系统来说这是一个新软件, 且为第一次安装

R (replace) 不是新软件, 不过会被重新 emerge (reemerged)

F (fetch) 该软件要求您先把源码手工地下载回来 (例如因为许可 (licencing issues) 的缘故)

U (update) 软件包已安装, 不过将被升级

dongwm@localhost ~ $sudo  emerge –update –deep world 注：world 文件不包含依赖关系. 当您 运行 emerge –update world 时, 仅有 world 文件中列出的那些软件包及其直接的 (immediate) 依赖包会被检查或, 必要的话, 会被更新. 如果您想让 emerge 检查所有的依赖关系 (包括依赖包的依赖包), 则请加多一个 –deep 参数:

解决安装redmine“undefined Method `requirement’ ”报错

Feb 15th, 2012

前言：在opensuse安装redmine出现一个比较可爱的报错，puppet面板安装也可能遇见，当创建表时：

dongwm@linux-b8lh:~/redmine/svn/branches/1.3-stable/config> !rake rake db:migrate RAILS_ENV=production (in /home/dongwm/redmine/svn/branches/1.3-stable)

rake aborted! undefined method `requirement’ for #<Rails::GemDependency:0xb71a2248>

Tasks: TOP => db:migrate => environment (See full trace by running task with –trace)

解决：修改gem_dependency.rb，一般在你安装的rails库目录下，我的是/usr/lib/ruby/gems/1.8/gems/rails-2.3.14/lib/rails/gem_dependency.rb

注释掉81行 #gem self.name, self.requirement # <  1.8 unhappy way 修改为：

gem self.name, :version => (self.respond_to?(:requirement) ? self.requirement : self.version_requirements)

Pymongo教程（二）

Feb 6th, 2012

1  移除子文档：

user_doc = { “username”:”foouser”, “emails”:[ { “email”:”foouser1@dongwm.com”, “primary”:True }, { “email”:”foouser2@dongwm.com”, “primary”:False }, { “email”:”foouser3@dongwm.com”, “primary”:False } ] } dbh.users.insert(user_doc, safe=True) dbh.users.update({“username”:”foouser”}, {“$pull”:{“emails”:{“email”:”foouser2@dongwm.com”}}}, safe=True)   #移除 “foouser2@dongwm.com”子文档

2 增加子文档

new_email = {“email”:”fooemail4@dongwm.com”, “primary”:False} dbh.users.update({“username”:”foouser”}, {“$push”:{“emails”:new_email}}, safe=True)   #使用$push增加

3 修改子文档

user_doc = { “username”:”foouser”, “emails”:[ { “email”:”foouser1@dongwm.com”, “primary”:True }, { “email”:”foouser2@dongwm.com”, “primary”:False }, { “email”:”foouser3@dongwm.com”, “primary”:False } ] } dbh.users.insert(user_doc, safe=True) dbh.users.update({“emails.email”:”foouser2@dongwm.com”}, {“$set”:{“emails.$.primary”:True}}, safe=True)   #设置子文档属性 “primary”为真

4 使用索引加快查找

这里有个文档

user_doc = { “username”:”foouser”, “emails”:[ { “email”:”foouser1@dongwm.com”, “primary”:True }, { “email”:”foouser2@dongwm.com”, “primary”:False }, { “email”:”foouser3@dongwm.com”, “primary”:False } ] }

dbh.users.insert(user_doc)

正常查找使用：

user = dbh.users.find_one({“emails.email”:”foouser2@dongwm.com”})

使用索引（根据mail.mail）：

dbh.users.create_index(“emails.email”)

注：简历索引比较耗时也增加系统负荷可以使用将其放在后台：dbh.users.create_index(“emails.email”, background=True)

5 删除索引

dbh.users.create_index(“username”, name=”username_idx”) dbh.users.drop_index(“username_idx”) 6 复合索引（简单的理解就是在where条件中字段用索引，如果用多字段就用复合索引）

dbh.users.create_index([(“first_name”, pymongo.ASCENDING), (“last_name”, pymongo.ASCENDING)])

注：删除索引方法：dbh.users.drop_index([(“first_name”, pymongo.ASCENDING), (“last_name”, pymongo.ASCENDING)])

或者：

dbh.users.create_index([ (“first_name”, pymongo.ASCENDING), (“last_name”, pymongo.ASCENDING) ], name=”name_idx”)  #给索引使用一个名字

7 geospatial（地理位置）索引

这个说来比较复杂：可以关注http://blog.nosqlfan.com/html/1811.html以及mongodbCTO的个人博客http://www.snailinaturtleneck.com/blog/
